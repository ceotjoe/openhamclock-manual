📻 WSJT-X / FT8

Feature

Live FT8, FT4, JT65, and WSPR decodes from WSJT-X or JTDX — plus PSKReporter coverage maps and RBN skimmer reports — all in real time.

A Quick Primer on Weak-Signal Modes

WSJT-X is a free program by Nobel laureate Joe Taylor, K1JT, that implements a family of highly efficient digital modes designed to make contacts in conditions where voice or CW would be impossible. The most popular are:

ModePeriodSensitivityBest use
FT8 15 seconds –21 dB SNR DX, casual contacts, POTA/SOTA; currently the most popular HF mode worldwide
FT4 7.5 seconds –17 dB SNR Contests — faster exchanges than FT8
JT65 60 seconds –25 dB SNR EME (moonbounce), very long-distance HF
WSPR 2 minutes –28 dB SNR Propagation beaconing and mapping — no QSOs, just signal reports

OpenHamClock receives the decoded messages from WSJT-X in real time and displays them as a live scrolling decode list — the same decodes you see in WSJT-X's own window, but integrated into your dashboard alongside DX spots, space weather, and the world map.

Local UDP Setup

If WSJT-X and OpenHamClock run on the same machine, or on machines on the same LAN, use the direct UDP connection. This is the simplest and most reliable option — no extra software, no accounts, sub-second latency.

  1. Enable WSJT-X integration in OpenHamClock

    In your .env file, set:

    .env 
    WSJTX_ENABLED=true
WSJTX_UDP_PORT=2237

    Restart the server after saving.

  2. Configure WSJT-X to send UDP packets

    In WSJT-X: File → Settings → Reporting tab.

    • Enable UDP Server
    • Set UDP Server hostname to the IP address of the machine running OpenHamClock (use 127.0.0.1 if it is the same machine)
    • Set UDP Server port number to 2237
    • Enable Accept UDP requests
  3. Verify the connection

    Start a WSJT-X decode session. Decodes should appear in the OpenHamClock WSJT-X panel within seconds. The panel header shows a green dot when receiving data.

WSJT-X on a different LAN machine

If WSJT-X runs on your shack PC and OpenHamClock runs on a Raspberry Pi, set the UDP destination in WSJT-X to the Pi's IP address (e.g. 192.168.1.100). Make sure the Pi's firewall allows UDP port 2237. Everything else is the same.

Cloud Relay Agent

If OpenHamClock runs in the cloud (Railway, VPS, etc.) it cannot receive raw UDP packets from your local WSJT-X — the internet is in the way. The wsjtx-relay agent solves this: it runs on your shack PC, captures WSJT-X UDP decodes locally, and forwards them to your cloud OpenHamClock instance over secure HTTPS.

  1. Set a relay key in your cloud .env
    .env (cloud instance) 
    WSJTX_ENABLED=true
WSJTX_RELAY_KEY=my-secret-relay-key   # any random string

    Redeploy the cloud instance after saving.

  2. Download wsjtx-relay to your shack PC

    Get the latest binary from the releases page: wsjtx-relay-win.exe, wsjtx-relay-macos, or wsjtx-relay-linux.

  3. Run the relay agent
    bash 
    $ ./wsjtx-relay \
    --target https://your-instance.railway.app \
    --key my-secret-relay-key \
    --listen-port 2237

    Configure WSJT-X to send UDP to 127.0.0.1:2237 as normal. The relay intercepts packets locally and forwards them to the cloud.

PSKReporter

PSKReporter is a global crowdsourced network of software-defined radio receivers that automatically report every digital mode signal they decode — FT8, FT4, JT65, PSK31, WSPR, and more. It is one of the best propagation analysis tools available to amateurs.

OpenHamClock shows PSKReporter data in two tabs:

TX tab Who is hearing your signal right now. Every station that decoded your callsign in the last 15 minutes appears on the map with a line from your QTH to theirs. This is the best real-time propagation check — is your signal actually reaching Europe? Asia? South America?
RX tab Who you are hearing — all callsigns your receiver has decoded, plotted on the map. Useful for seeing the reach of your antenna and receiver setup.

Data arrives via MQTT WebSocket in near-real-time (sub-second updates) when the MQTT connection is available, or falls back to the HTTP API polled every 2 minutes. No configuration needed — PSKReporter uses your callsign from .env automatically.

Reverse Beacon Network (RBN)

The Reverse Beacon Network is a global network of automated CW and RTTY receivers (skimmers) that listen 24/7 and report every CQ call they hear. Think of it as a DX cluster for CW — the difference is that RBN spots are generated automatically by software, not posted manually by operators.

OpenHamClock shows RBN skimmer locations on the world map and lists recent spots in the RBN panel. This is particularly useful for:

  • CW contesting — see which skimmers heard your CQ and at what SNR, confirming your signal is getting out.
  • Propagation checking — see active CW stations at a glance without wading through a full DX cluster feed.
  • DXpedition spotting — rare expeditions operating CW are spotted immediately by multiple skimmers worldwide.

WSPR Heatmap

WSPR (Weak Signal Propagation Reporter, pronounced "whisper") is a beacon mode where stations transmit a low-power signal containing their callsign, grid square, and power level. Receiving stations report to WSPRnet, building a global database of who heard whom at what time.

The OpenHamClock WSPR heatmap overlays this data on the world map as a colour-coded signal density layer — brighter areas indicate more WSPR reports, giving an instant visual snapshot of HF propagation conditions on the selected band. It is particularly beautiful to watch during a grey-line pass: you can literally see the propagation paths light up as the terminator sweeps across the globe.
